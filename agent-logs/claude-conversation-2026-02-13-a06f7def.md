## The Problem

It's not Handle returns that are the issue — those work fine. The problem is that **collection typedefs** (like `Poly_ListOfTriangulation`) fall through a gap in the type system.

### How Handle returns work (successfully)

Methods like `BRep_Tool::Triangulation()` returning `const Handle(Poly_Triangulation)&` work because `Poly_Triangulation` is a parsed class — it's in `all_classes`, so the filter passes. Many Handle-returning methods are generated (`BRep_Tool_surface_face`, `Geom_Curve_reversed`, etc.).

### Why Triangulations() is filtered

`BRep_Tool::Triangulations()` returns `const Poly_ListOfTriangulation&`, where `Poly_ListOfTriangulation` is a typedef:
```cpp
typedef NCollection_List<Handle(Poly_Triangulation)> Poly_ListOfTriangulation;
```

The filtering chain in `bindings.rs`:

1. Return type parsed as `Type::ConstRef(Type::Class("Poly_ListOfTriangulation"))`
2. `type_uses_unknown_type()` checks if `"Poly_ListOfTriangulation"` is in `all_classes`
3. **It's not** — typedefs aren't parsed as classes
4. Method filtered out as having an unknown type

The irony: `Poly_ListOfTriangulation` IS declared as an opaque type in `ffi.rs` (via the collections system), but the method filtering code doesn't know about it. Collections are added to the CXX bridge but never added to the `all_classes` set that the filter checks.

### Scope of the problem

This affects **any method returning a collection typedef** that isn't already in the hardcoded `known_collections` list. And even for known collections (like `TopTools_ListOfShape`), methods returning them by reference would likely be filtered for the same reason — though I'd want to verify that.

**Option A — Make filtering collection-aware (targeted fix):** Thread the known collection names into `TypeContext` and treat them as known types in `type_uses_unknown_type()`. Collection typedefs would pass the filter just like parsed classes.

**Option B — Add Poly_ListOfTriangulation to known_collections + Option A:** Also add `Poly_ListOfTriangulation` to the known collections list so it gets iterator wrappers, then apply Option A so methods returning it aren't filtered.

**Option C — Parse typedefs into all_classes (architectural fix):** Make the parser recognize NCollection typedefs as synthetic classes and add them to `all_classes`. This automatically supports any future collection type without hardcoding.
